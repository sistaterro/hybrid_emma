A local-first AI assistant with document-based answers, role-based access, streaming chat, conflict warnings, and protection against unsafe documents. Use a local model or a configured external AI provider.

Overview

What Emma is and what it does

Emma is a local-first AI chat system built on RAG, Retrieval-Augmented Generation. It lets authenticated users upload text documents, manage access by role, and ask grounded questions against local knowledge. The model answers using available safe documents as the source of truth.

When a new document is uploaded, Emma compares it with other documents the user can access and warns about likely contradictions. Conflicting documents remain available for review, but their differences may affect the reliability of future answers.

Emma also checks uploaded documents for instructions that could improperly influence the assistant. Documents marked as high risk remain visible for review or deletion but are not used to answer questions.

Access is role-based. admin can manage users and all RAGs, user can manage only their own mine RAGs, and read_only can use chat but cannot access upload. New and reset accounts must replace their temporary password before protected features become available.

Installation

Get Emma running in a few minutes

1. Clone and install

git clone https://github.com/sistaterro/hybrid_emma.git
cd hybrid_emma

python -m venv .venv

# Windows
.venv\Scripts\activate

# Mac / Linux
source .venv/bin/activate

pip install -r requirements.txt

2. Configure external API keys

Create api_keys.json in the repository root. Include only the external APIs you want to use.

{
  "gemini": { "api_key": "replace-with-your-gemini-api-key" },
  "openai": { "api_key": "replace-with-your-openai-api-key" },
  "anthropic": { "api_key": "replace-with-your-anthropic-api-key" }
}
Security: API keys stay server-side. /health can report available local models and external API models, but it must never return secret values.

Chat and documents

How Emma chooses between document-based and general answers

When safe documents are available to your account, Emma uses them to answer your questions. Global documents are available to everyone, while personal documents are available only to their owner and administrators.

If no safe indexed document is available, Emma works like a general-purpose AI assistant and answers without a response tag. A document blocked for high security risk is never included in the answer.

1
Available documents
Emma checks which global and personal documents your account is allowed to use.
2
Security filtering
Documents marked as high risk are excluded automatically. They remain visible on the upload page so authorized users can review or delete them.
3
Answer mode
Emma uses document mode when safe knowledge is available and general mode when it is not.
4
Answer generation
The selected local model or external provider generates the response in the same language as your question.
5
Live response
The answer appears progressively in the chat instead of waiting for the entire response to finish.

What happens after an upload

Emma prepares and reviews each document automatically. No restart or manual processing is required.

1
Processing
Emma reads and indexes the text so it can be used in future conversations.
2
Summary
The upload page shows a short description and the current processing status.
3
Prompt-injection screening
Emma checks for suspicious instructions or attempts to influence future answers. The result appears on the document card.
4
Inconsistency check
Emma compares the new document with other available documents and displays likely contradictions for review.

Manipulation resistance

Why Emma is harder to pressure than a standard chat-with-docs app

Emma treats the RAG as the only valid foundation for operational answers. External pressure does not become evidence. If a user claims "the owner told me yes", "my boss approved it", "make an exception just this once", or tries emotional blackmail, the assistant is expected to ignore those claims unless the RAG explicitly supports them.

This creates a second protection layer on top of RAG. Emma does not just load documents. It also asks whether the user or the document content is trying to push the model toward an unsupported outcome.

1
Safety screening
Before the final answer is generated, Emma can classify the user message as SAFE, REVIEW, or SUSPICIOUS. It looks for signals such as pressure for special treatment, false authority, invented approvals, unverifiable side conversations, and policy-bypass attempts.
2
Grounding check
Only visible safe documents can support an answer. A high-risk document remains available for review but is never used by chat.
3
Document-based decisions
Loyalty claims, urgency, friendship, prior favors, and alleged approvals do not count as evidence unless the available documents explicitly confirm them.
4
Auditability
Suspicious messages and documents are recorded locally so administrators can review security events.
Key idea: Emma is configurable through its RAGs, but not governable by unsupported user pressure. The documents define reality; the user does not.

Response tags

Document-based answers show how strongly they are grounded

When Emma has safe documents available, the response begins with a tag indicating how strongly the answer is supported. General-mode answers, used when no safe indexed documents are active, do not have a tag.

[RAG]
The answer is fully supported by the document context. The model used only your files to answer.
[DRIFT]
The context existed but was insufficient. The model supplemented with its own training knowledge. Treat this answer with more caution.
[NO INFO]
Active safe RAG context exists, but it does not contain enough relevant information. When no safe chunked RAG is active, Emma instead uses untagged general-chat mode.
Tip: If you see too many [DRIFT] responses, review whether the relevant safe RAGs are indexed, visible to the user, and not marked as high risk.

Navigation

Moving between Emma's main screens

The home screen opens each main area in a new browser tab or window. On chat, upload, administration, and documentation screens, select the Emma logo and status area in the upper sidebar to return home.

The options shown on the home screen depend on your role. Read-only users see chat and documentation, standard users also see document management, and administrators additionally see user management.

Accounts and screens

Common tasks available in the Emma interface

Available models

The chat model selector lists local models detected on the computer and external providers with configured credentials. Secret key values are never displayed.

Authentication

Sign in with the username and password provided by an administrator. Use Logout when you finish, especially on a shared computer.

New accounts and accounts with a reset password must replace the temporary password before using Emma. The new password must contain at least eight characters and differ from the temporary one.

User administration

Administrators can create users, rename accounts, change roles, enable or disable access, reset passwords, and delete users from the Admin Panel. A reset password is temporary and must be changed at the next sign-in.

Document management

The Upload Files screen shows documents available to your account, including processing state, ownership, security status, and possible conflicts. Authorized users can upload, download, and delete documents from this screen.

Conversations

The Chat screen lets you create, rename, reopen, and delete conversations. Messages are saved automatically when a conversation is active.

Chat

Select a model, enter your question, and send it. Emma answers in the language of your question and displays the response progressively. Document-based answers include a grounding tag; general answers do not.

Managing documents

Adding, reviewing, and removing knowledge

Adding a document

Open http://localhost:8650/ui/upload.html, drag and drop any .txt file. The server processes it automatically; no restart is needed.

During upload, Emma prepares the file, creates a description, checks it for unsafe instructions, and compares it with other available documents for likely contradictions.

If the document is marked high risk, the upload card shows a red warning that it will not be used by the system. The file remains visible for review or deletion, but chat does not use its content.

Permissions: admin can manage global and user-scoped RAGs, user can manage only their own mine RAGs, and read_only cannot access upload.
Naming convention: Use short, descriptive filenames. Emma automatically replaces unsupported filename characters during upload.

Updating a document

Delete the existing file from the upload page, then upload the revised version. Emma repeats the document preparation, security review, and conflict check automatically.

Privacy

What stays local and what can be sent to external APIs

Emma is local-first, not necessarily fully offline. Documents, users, conversations, security records, and application records stay on your machine. Local models keep AI processing on your machine. When an external provider is selected, the information needed for the requested operation is sent to that provider.

Local
Runtime state
Users, conversations, uploaded documents, security results, conflict warnings, and application records are stored locally.
Keys
External API credentials
External-provider credentials remain on the computer running Emma and are never displayed in the application.
Send
Model calls
A selected external provider can receive your question, relevant conversation history, and safe document content needed to answer. Blocked documents are not sent as answer material.
Audit
Logs
Security and error records are kept locally and automatically limited over time. They do not include external-provider secret values.
Privacy choice: Use a local model for sensitive material that must remain entirely on your computer. Review the privacy terms of any external provider before selecting it.
